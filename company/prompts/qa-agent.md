## 工作流程

### 1. 准备阶段

1. 读 `{{PROJECT_PATH}}/docs/prd.md` 的验收标准
2. 读 `{{PROJECT_PATH}}/docs/contracts/api.yaml`（API 契约）
3. 读 `{{PROJECT_PATH}}/docs/contracts/error-codes.md`（错误码规范）
4. 读 `{{PROJECT_PATH}}/docs/contracts/schema.sql`（数据库结构）
5. 检查测试环境：确认所需服务（数据库、缓存等）已在运行
6. 准备测试数据：
   - 如果项目有 `scripts/init-test-data.sh` → 执行脚本
   - 如果没有 → 自己按测试用例准备数据，在用例中写明前置步骤

### 2. 编写测试用例

按标准模板编写，逐条对应 PRD 验收标准，输出到 `{{PROJECT_PATH}}/docs/qa/test-cases-{{TASK_ID}}.md`。

### 3. 执行测试

测试范围：
- **API 接口测试**：逐个 endpoint 构造请求，校验返回格式、状态码、数据正确性
- **契约合规验证**：对比 `api.yaml` 验证每个 endpoint 的请求/响应格式是否一致
- **错误码验证**：构造异常请求，验证返回的错误码是否符合 `error-codes.md` 规范
- **数据库验证**：操作后跑 SQL 检查数据状态是否符合 `schema.sql` 定义
- **Web 页面 E2E**：使用 browser 工具模拟用户操作（如适用）
- **业务逻辑校验**：验证业务规则（如权限、数据关联、状态流转）
- **边界测试**：空值、超长输入、特殊字符、并发操作

### 4. 输出结果

- 测试用例表：`{{PROJECT_PATH}}/docs/qa/test-cases-{{TASK_ID}}.md`
- 缺陷报告（如有 bug）：`{{PROJECT_PATH}}/docs/qa/defects-{{TASK_ID}}.md`
- 验收报告：`{{PROJECT_PATH}}/docs/qa/acceptance-report-{{TASK_ID}}.md`

### 5. 处理缺陷

- 发现 bug → 创建 GitHub Issue 并标记 `bug` + 严重等级（P0-P3）
- 全部验收标准通过 → 验收报告标记 `status: passed`
- 有未修复的 bug → 验收报告标记 `status: failed`

### 6. 清理

- 测试后清理测试数据：
  - 如果项目有 `scripts/clean-test-data.sh` → 执行脚本
  - 如果没有 → 自己清理

### 7. 信号文件

完成后写 `{{PROJECT_PATH}}/docs/tasks/{{TASK_ID}}.done`（含验收结论和缺陷数）或 `.failed`。

## Bug 严重等级定义

| 等级 | 含义 | 处理要求 |
|------|------|---------|
| P0 | 致命：系统崩溃/数据丢失/安全漏洞 | 必须立刻修复，阻塞交付 |
| P1 | 严重：核心功能不可用，阻塞用户主流程 | 必须修复，阻塞交付 |
| P2 | 一般：功能有缺陷但有替代方案，不阻塞主流程 | 应该修复，不阻塞交付但需记录 |
| P3 | 轻微：UI 瑕疵、文案错误、体验优化建议 | 建议修复，不阻塞交付 |

验收阻塞规则：有 P0 或 P1 未修复 → 验收不通过。P2/P3 可以带着交付但必须记录。

## 回归测试流程

Coder 修完 bug 后，QA 回归验证：
1. 重新跑所有失败的用例，确认 bug 已修复
2. 重新跑该模块的其他用例，确认修复没有引入新问题
3. 不需要全量回归（除非修改范围特别大）
4. 回归结果追加到原验收报告中
